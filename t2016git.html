git

The Stupid Content Tracker

  • git is NOT GitHub.
  • GitHub is 'server git'

Why git over another VCS?

  • Branching and Merging
  • ***FAST***
  • Powerful, and Flexible
  • Free (as in freedom, and as in beer)

Why git over another client?

  • git is the 1st party front-end
  • git is the way it was designed to be experienced.
  • Learning git is the easiest way to understand what's going on.
  • git gives you ultimate, uninhibited access to your repository.

git Vocabulary

git Branching

  • We'll discuss what we're doing in a bit
  • Let's create a new feature, but we know it's going to be buggy
    • Let's create it in an isolated environment, a branch!

Create a feature branch

1 # Create a myfeature branch
2 git branch myfeature
3 # Move to myfeature branch
4 git checkout myfeature
5 
6 # Let's find out where we are
7 git status
8 # > On branch myfeature
9 # > nothing to commit, working tree clean

Add a new 'feature'

1 echo "My new feature!" >> file.txt
2 git add file.txt
3 git commit -m 'My new feature is very cool, but buggy!'

There's a bug in the released version!

  • Let's patch the version everyone is on, the master release!
1 # Switch back to the master branch
2 git checkout master
3 git status
4 # Look at the contents of file.txt at this point!
5 
6 echo "My critical bugfix!" >> file.txt
7 git add file.txt
8 git commit -m "Critical bugfix number 1!"

What just happened?

  • We first created our test file, which is our 'release version'
  • Then we created a feature branch, and made some 'buggy changes'
  • We then realized we need to fix a bug in the 'release version'
    • We switched back to release, and 'fixed the bug'

Merging

  • Brings divergent branches together
  • Let's release our feature by putting it into the master branch

git Remotes

  • How do we share our code with others?
  • Branching is great for me, but what if I want to let my friends work on a branch?

Let's Play With Remotes

1 # let's start in a common state
2 git checkout master
3 
4 # let's make a copy for 'our friend to have'
5 cd ..
6 # Clone from the repo folder to the repo2 folder
7 git clone repo/ repo2/
8 cd repo2
9 git status
10 # > On branch master
11 # > Your branch is up-to-date with 'origin/master'.
12 # > nothing to commit, working tree clean
13 git log --oneline --abbrev --graph --all
14 # * 8f12721 Merge branch 'myfeature'
15 # |\
16 # | * 2716a3b Add a new feature
17 # * | 625aaf5 Critical bugfix
18 # |/
19 # * cbbd8d9 one small change

Let's Make a Change On 'our computer'

1 # Move into 'our repo'
2 cd ../repo
3 git status
4 
5 # Edit file.txt with our changes
6 echo "A change I made on my computer" >> file.txt
7 
8 git add file.txt
9 git commit -m 'Added files from my computer.'

Let's Give This Change To Our Friend

1 # Pull new changes from repo's master into repo2's master
2 cd ../repo2
3 git pull ../repo master
4 # > From ../repo
5 # > * branch master -> FETCH_HEAD
6 # > Updating 8f12721..2b05fa9
7 # > Fast-forward
8 # > file.txt | 1 +
9 # > 1 file changed, 1 insertion(+)

Shortcuts to make remotes faster

1 # Add a shortcut to interact with ../repo2
2 cd ../repo
3 git remote add repo2 ../repo2
4 # same as git pull ../repo2 master
5 git pull repo2 master
6 # > From ../repo2
7 # > * branch master -> FETCH_HEAD
8 # > Already up-to-date.
9 
10 # List existing remotes
11 git remote -v
12 # > repo2 ../repo2 (fetch)
13 # > repo2 ../repo2 (push)
14 
15 # man git-remote for more options, like
16 # git remote set-url repo2 ../repo2
17 # git remote rm repo2
18 # git remote show repo2

A note about git fetch

  • git fetch updates what we see in a remote without changing our files.
  • git pull updates our files with changes
  • git fetch stores those changes made without touching our state.

git refs

  • git has a suite of commands for interacting with history.
    • To use them, you will need to understand refs
  • A ref is anything that identifies a commit&x2026;

Examples of Refs

  1. A Commit Hash (eg: cbbd8d9a5e1c64f84)
    • You can shorten with the first few chars
  2. Special: HEAD
    • A Ref to the commit your repository is currently on
  3. Branch Names
    • master, myfeature
  4. Remotes
    • <remote>/<branch>
    • repo2/master, origin/master
  5. Tags
    • Aliases you can give to commits

Ref Modifiers

  • Using HEAD as an example.
  • HEAD~; One commit before HEAD
  • HEAD~~; Two commits before HEAD
  • HEAD~3; Three commits before HEAD
  • HEAD^<N>; Choose the <Nth> parent if we're on a merge.

Using Refs

1 # Show's the current commit, same as git show
2 git show HEAD
3 
4 # Shows the commit that's one old
5 git show HEAD~
6 
7 # Shows the commit thats two commits older than repo2's master
8 git show repo2/master~~

Quickly See Files in a Ref

1 git checkout <ref>
2 
3 # Does nothing, HEAD is our current ref
4 git checkout HEAD
5 
6 # See the commit before head
7 git checkout HEAD~
8 # Warning about detached HEAD
9 
10 # checkout to 2 commits before the remote's master branch.
11 git checkout repo2/master~~
12 
13 # We also use checkout to switch branches (we can edit safely now)
14 git checkout master

Change Where a Branch 'Points To'

1 # Git reset has many uses
2 # Run git status yourself!
3 
4 # Move our branch to the previous commit, but don't change our files.
5 # Any diff is staged
6 git reset --soft HEAD~
7 # Go back to where we were
8 git pull repo2 master
9 
10 # Move our branch to the previous commit, and change our files
11 # No diff is present (since we change files)
12 git reset --hard HEAD~
13 git pull repo2 master
14 
15 # Like git reset --soft, but don't stage our files either.
16 git reset --mixed HEAD~
17 git pull repo2 master
18 # Complaint about unstaged files
19 
20 # HEAD is the default if no ref is supplied
21 # This resets our files to head, trashing our 'unstaged work'
22 git reset --hard
23 git pull repo2 master

Find the Diff Between Two Commits

1 # Finds the difference between one commit ago and the current commit
2 git diff HEAD~ HEAD
3 
4 # Finds the difference between myfeature and master
5 git diff myfeature master
6 
7 # Show unstaged changes
8 git diff
9 
10 # Show only staged changes
11 git diff --staged

Pull Request

  • A PR is a request for a maintainer to pull code into a repo you don't have access to
  • First fork the repo (this is a clone from a github repo to a github repo you have write access to)
  • Then clone your fork (you will now be able to commit/push to this fork)
  • Submit a PR from your fork into the target repo
  • Once accepted, the maintainer pulls code from your github fork to the main github repo.
